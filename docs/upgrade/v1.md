# Upgrading to Hooka v1

This guide covers the runtime and schema expectations for the `1.0.0-rc.1` line.

## Required env changes

Server deployments now require:

- `HOOKA_WEBHOOK_SECRET`
- `HOOKA_ADMIN_TOKEN`

Hooka also resolves targets from:

- `HOOKA_TARGETS_PATH`
- default: `.hooka/targets.json`

## Runtime changes

- read/admin APIs are protected by `Authorization: Bearer <HOOKA_ADMIN_TOKEN>`
- `/api/targets` and `/api/targets/:id` expose configured targets
- `/api/runs/:id/retry` retries terminal runs by enqueueing a new run
- `/api/events/stream` emits SSE updates for run events and worker heartbeats
- target-based webhooks are supported alongside existing generic `taskId + input` webhooks

## Run model changes

Hooka now records additional run metadata including:

- `targetId`
- `maxAttempts`
- `nextRetryAt`
- `lastErrorCode`

Supported statuses now include:

- `dead-lettered`

Retryable failures are requeued with backoff. Exhausted retry budget or recovered stuck runs can end in `dead-lettered`.

## Compatibility notes

- existing generic webhook payloads remain valid
- existing WordPress compatibility webhook route remains valid
- worker image taxonomy and preset aliases are unchanged
- official support remains single-node self-host with SQLite
