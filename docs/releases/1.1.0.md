# Hooka `1.1.0`

`1.1.0` is the first feature release after the stable `1.0.0` cut. It keeps
Hooka's single-node, self-hosted, SQLite-backed operating model while making
multi-preset worker deployments and target-backed webhook flows safer to run in
production.

## Included in `1.1.0`

- capability-aware worker claiming, so workers only claim queued runs they are
  able to execute
- target-backed compatibility webhooks through optional `targetId` support in
  WordPress and TrailBase-compatible payloads
- generic webhook smoke testing with `hooka webhook send` for raw JSON bodies
  signed with HMAC
- artifact readiness checks with recursive quiet-period support and
  required-file validation
- target execution controls with per-target concurrency limits
- a workflow chaining RFC for future `verify -> deploy -> smoke -> notify`
  orchestration
- `rclone-sync` as an active worker preset with shared-volume copy tasks
- hardened runtime/operator workflows, including improved cleanup, OpenAPI,
  auth, rate limiting, executor timeout, and Docker manifest validation paths
- automated GHCR prerelease image cleanup after stable tag publication

## Compatibility notes

- Existing generic webhook payloads remain supported.
- Existing WordPress and TrailBase compatibility webhook payloads remain
  supported without `targetId`.
- `targetId` is optional for compatibility adapters; when present, the adapter
  reuses the target policy path instead of introducing producer-specific server
  logic.
- Targets default to one concurrent run unless `maxConcurrentRuns` is set.
- Artifact readiness can still use the previous marker and quiet-period modes.
  `requiredFiles` and recursive quiet-period checks are additive.
- Workers preserve queue visibility for task IDs not known to the current
  registry, so custom or future task definitions are not silently hidden from
  compatible workers.

## Operational notes

- Mixed preset workers can now share the same `HOOKA_DB_PATH` with less risk of
  the wrong worker claiming a run.
- Target-backed deploy and sync flows can use `maxConcurrentRuns: 1` to avoid
  overlapping mutations against the same target.
- Shared-volume artifacts can be validated with required files and recursive
  quiet-period checks before worker execution starts.
- Operators can smoke-test any signed webhook route by passing the exact raw
  JSON body to `hooka webhook send`.
- The workflow chaining RFC is documentation-only. It does not add runtime DAG
  execution yet.

## Deferred after `1.1.0`

- workflow chaining runtime implementation
- richer dashboard-first target authoring
- producer-specific secret rotation
- multi-node/Postgres support
